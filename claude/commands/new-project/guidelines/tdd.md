# TDD（テスト駆動開発）ガイドライン

実装前にテストを書き、テストを通す実装を書く。手戻りを減らしトークンを節約するための開発手法。

## サイクル

```
1. Red   — 失敗するテストを書く（実装はまだない）
2. Green — テストが通る最小限の実装を書く
3. Refactor — 振る舞いを変えずにコードを整理する
```

1サイクルの単位は小さく保つ。1関数・1エンドポイントが目安。

## tester エージェントの呼び出しタイミング

| タイミング | モード | 呼び出し方 |
|---|---|---|
| 実装前（TDD） | Red フェーズ：失敗するテストを書く | `tester`を実装前に呼ぶ（TDDモードと明示） |
| 実装後（補完） | 実装済みコードのテストを追加する | `tester`を実装後に呼ぶ（通常モード） |

## テストを先に書く対象

以下は TDD の効果が高い。必ずテストファーストにすること。

- ビジネスロジック（計算・バリデーション・状態遷移）
- API エンドポイント（正常系・異常系・認証）
- DB 操作（CRUD・トランザクション・制約）

以下は実装後でよい。

- UI コンポーネント（スナップショット等）
- 外部サービスの薄いラッパー

## Red フェーズのルール

- テストは実行して**必ず失敗を確認**してから実装に進む（未実装で green になる場合はテストが壊れている）
- テスト名は「[条件]のとき[期待結果]」の形式にする
- 1テストにアサーションは原則1つ

## Green フェーズのルール

- テストを通す**最小限の実装**だけ書く。過剰な実装をしない
- すべての既存テストが引き続き green であることを確認する

## Refactor フェーズのルール

- テストが green の状態を維持しながら整理する
- DRY・命名・構造の改善に集中する。振る舞いは変えない
