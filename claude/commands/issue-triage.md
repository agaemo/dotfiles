## 役割

issue の内容を読んで「どう対応するか」を判断し、ユーザーに確認を取ってから実行する。
判断だけして終わるのではなく、承認されたら修正・PR作成まで担う。

## 入力

```
/issue-triage {issue番号}
/issue-triage {issue番号} {owner/repo}
```

`{owner/repo}` 省略時は `gh repo view --json nameWithOwner -q .nameWithOwner` で現在のリポジトリを取得する。

## プロセス

### ステップ 1：issue を読む

```bash
gh issue view {number} --json number,title,body,labels,comments
```

### ステップ 2：内容を分析する

以下の観点で評価する：

| 観点 | 確認内容 |
|------|----------|
| **カテゴリ** | バグ / コードスメル / 型・ドキュメント不足 / デッドコード / 機能追加 |
| **影響範囲** | 変更が必要なファイル数・モジュール数 |
| **リスク** | 動作変更を伴うか / テストが必要か / ビジネスロジックの理解が必要か |
| **複雑度** | 自己完結しているか / 他の変更との依存があるか |

### ステップ 3：対応方針を提示してユーザーに確認する

分析結果をもとに、以下のいずれかを推奨する：

#### A. 直接修正（シンプル・安全）

条件：1〜3ファイル以内、動作変更なし、リバート容易

```
【推奨】直接修正

対象：{ファイルパス:行番号}
修正内容：{何を・どう変えるか}
リスク：低（{理由}）

修正してPRを作成しますか？ (y/N/wontfix)
```

#### B. 計画が必要

条件：複数ファイル連動 / ロジック変更 / アーキテクチャ判断が必要

```
【要検討】計画が必要

理由：{なぜシンプルに直せないか}
検討事項：
  - {論点1}
  - {論点2}

推奨する進め方：{具体的な手順や設計方針の提案}

この方針で進めますか？ (y/N/wontfix)
```

#### C. wontfix

条件：意図的な設計 / コスト対効果が低い / フレームワーク制約による回避策

```
【wontfix推奨】

理由：{なぜ修正不要か}

issueをcloseしますか？ (y/N)
```

### ステップ 4：承認されたら実行する

**A（直接修正）が承認された場合：**

1. ブランチを切る：`git checkout -b fix/{issue番号}-{brief-slug}`
2. 修正を適用する
3. ビルド・テストを実行する（`validateCommand` があれば使う、なければ `tsc --noEmit` など適切なコマンドを判断）
4. 問題なければコミット・PR作成：

```bash
gh pr create \
  --title "fix: {タイトル}" \
  --body "$(cat <<'EOF'
## Summary

{修正内容を1文で}

Closes #{issue番号}
EOF
)"
```

**B（計画が必要）が承認された場合：**

承認された方針に従って実装する。複数PRに分ける必要があれば分割して進める。

**C（wontfix）が承認された場合：**

```bash
gh issue close {number} --reason "not planned" --comment "意図的な設計のためcloseします。{理由}"
```

## ルール

- 修正前に必ずユーザーの承認を取ること。サイレントに変更しない
- 承認なしに `wontfix` でcloseしない
- 修正内容は最小限にとどめる。issueのスコープを超えた変更はしない
- ビルドが通らない場合は変更を戻してその旨を報告する
