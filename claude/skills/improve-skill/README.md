# improve-skill

スキルファイル（`SKILL.md`）を静的解析・実行シミュレーションで改善するスキル。
実プロジェクトを動かさずに低トークンで反復改善できる。

## 使い方

```
/improve-skill <ファイルパス>        # 単体
/improve-skill path/a.md path/b.md  # 複数
/improve-skill all                  # .claude/skills/ 全件
```

## 6次元の静的評価

スキルの種別と書き込みの有無によって **[D][E] の評価内容が変わる**。

```
DETECT スキルの種別（[E] の評価方法）
  プログラム型: Agent ツール（サブエージェント）呼び出しを含む
  指示文型:    Claude への自然言語指示のみ

DETECT 書き込み・不可逆操作の有無（[D] の評価方法）
  あり: Write / Edit / Bash（副作用）を含む
  なし: Read / Glob / Grep / WebSearch のみ
```

| 次元 | 観点 | 何をチェックするか |
|------|------|--------------------|
| **[A] トークン効率** | コスト | 情報の重複。削除しても意味が変わらない説明・前置き。肥大化した SKILL.md を相互排他的なサブファイルに分割できないか。参照ファイルが実行開始時に一括ロードされる設計になっていないか（just-in-time 読み込みが望ましい）。 |
| **[B] 明瞭性** | 解釈ブレ | 命令が1通りにしか解釈できないか。変数名・ファイルパス・条件が文脈なしで特定できるか。`description` が「このスキルを呼ぶべき状況」を一文で判断できる具体性を持つか（description の質が自動呼び出し精度を直接決める）。重要なルールに `IMPORTANT` 等の強調があるか。 |
| **[C] 完全性** | 抜け漏れ | 各ステップに失敗時の処理があるか。条件分岐（IF/ELIF/ELSE）の全パスが網羅されているか。ステップ間のデータ受け渡しが明示されているか。ループ処理でイテレーションごとに行うことと全体で1回だけ行うことが区別されているか。 |
| **[D] ゲート設計 / スコープ純粋性** | 安全性 | **書き込みあり（ゲート設計）:** `WAIT_FOR` が不可逆操作の前にあるか。`PROHIBITED` でゲートの抜け道が塞がれているか。 **書き込みなし（スコープ純粋性）:** `description` で宣言した目的以外の操作をしていないか。予告なく副作用が発生しないか。 |
| **[E] エージェント・対話効率** | 効率 | **プログラム型:** サブエージェントのプロンプトに不要なコンテキストがないか。直列だが並列化できるステップがないか。二重作業がないか。 **指示文型:** `WAIT_FOR`（ユーザー確認）の回数が最小限か。複数の質問を1メッセージにまとめているか。スキップ条件・早期終了条件が定義されているか。 |
| **[F] Progressive Disclosure** | 段階的開示 | `description` だけでトリガー判断ができる具体性か（第1段階）。`SKILL.md` 本体を読めば実行情報が揃うか（第2段階）。参照ファイルが「必要になった段階で初めて READ する」設計か（第3段階以降）。肥大化した SKILL.md を分割できる余地がないか。 |

## Adversarial Trace

「コールドスタートのサブエージェントがこのスキルを初めて見る」ペルソナで、
意図的に誤解しやすい解釈でステップを読み、混乱・スキップ・誤動作が起きる箇所を探す。
各発見には **Root Cause**（なぜ起きるか）を付記し、ステップ4の提案根拠として引き継ぐ。

## 評価後のフロー

1. スコアと改善提案を表示（各提案に **改善次元** と **trade-off** を付記）
2. 全書き直し判定（2/5 以下が 3 次元以上ならユーザーに確認）
3. RECOMMEND: Pareto 改善（複数次元同時改善）を優先して上位 3 件を提示
4. ユーザーが適用する提案番号を選択（`all` / `1 3 5` / `skip`）
5. 適用 → READ で確認 → 報告
6. `/loop` 実行時は **Rejection buffer** により前回却下された提案の再生成を防ぐ

ループ制御・Rejection buffer は SkillOpt（arxiv:2605.23904）、Root Cause 診断・Pareto 改善は GEPA（arxiv:2507.19457）を参考に設計。
