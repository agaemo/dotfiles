# output-brief

```
SKILL_DIR     = このSKILL.mdが存在するディレクトリの絶対パス
RESEARCH_FILE = /tmp/output-brief-research.md  （調査結果の一時ファイル。使用後に削除する）
```

## 発動判断

以下のいずれかに該当する場合のみブリーフィングを行う。**全て指定済みなら即出力する（ステップ1〜3をスキップしてステップ4へ進む）。**

- 保存先（ファイル保存 / チャット表示）が未指定
- 出力形式（HTML / Markdown / CSV …）が未指定
- HTML でレイアウト・カラーが未指定
- コンテンツ内容・構成が不明
- データ・素材が無い、または不十分・曖昧で調査が必要

## フロー

### 質問の提示方法（共通ルール）

ユーザーへの確認・選択肢提示は `AskUserQuestion` ツールを使わず、**テキストの番号リスト**で行う:

```
[1]: <選択肢1>（必要なら補足）
[2]: <選択肢2>
[3]: <選択肢3>
```

複数の質問をまとめて聞く場合は、質問ごとに短い見出しを付けてから番号リストを並べる。
選択肢以外の自由記述を許す場合はリストの末尾に「その他: 自由に記入してください」を添える。
WAIT_FOR: ユーザーの返信（番号・選択肢のテキストいずれでも回答可）。

### 1. データ確認

指示から以下3点を把握する。**全て読み取れる場合はステップ2へ直行する。不明な点があれば番号リスト形式で1回にまとめて確認する:**

- 何を作るか（ファイル種別・内容）
- データ・素材はあるか（URL・ファイルパス・テキスト）
- 保存先の指定はあるか（ファイルとして保存 / チャット表示）。ファイル保存の場合は保存先パス（ファイル名含む）も確認する

**調査が必要な場合**

以下のいずれかに該当する場合は、ステップ2に進む前に `WebSearch` / `WebFetch` で調べる:

- データ・素材が指定されていない（「〜について調べて出力して」「〜を調査してまとめて」等）
- データ・素材はあるが、出力に必要な情報が不足している、または内容が曖昧（自分の知識だけでは出力内容の正確性に確信が持てない・指示文中の事実に最新情報の裏取りが必要・用語の定義が不明 等）

調査範囲が広く絞り込みが必要な場合や、情報源が乏しく方向性に迷う場合は、調査を始める前に番号リスト形式で対象・観点を確認する。
調査しても十分な情報が得られない場合は、その旨をユーザーに伝え、手がかり（URL・キーワード等）を追加でもらうか、不十分な情報のまま進めてよいか確認する。
調査結果は箇条書きやメモとして整理し、`Write` ツールで `RESEARCH_FILE` に保存する（Markdown形式）。
保存内容: 検索クエリ・参照URL・調査結果（箇条書き）・出典リスト。
Write に失敗した場合は調査結果をインメモリ（会話コンテキスト）で保持して続行する。
ステップ2以降では `Read` で `RESEARCH_FILE` を参照して内容を確認する。
手順・フロー・関係性など図解が理解を助ける内容が含まれる場合は、Mermaid図の追加も候補として検討し、ステップ2の確認に含める。

### 2. ブリーフィング

選択肢は「質問の提示方法（共通ルール）」の番号リスト形式で提示する（`AskUserQuestion` は使わない）。

#### 調査結果の確認（ステップ1で調査した場合）

`RESEARCH_FILE` を `Read` して調査内容を参照し、要約（出典・主要な事実を箇条書き）で提示する。「この内容で出力してよいか」「不足や見当違いがあれば指摘してほしい」と確認する。
出典（URL）を出力本文にも参考リンクとして含めるか、不要かも合わせて確認する。
調査結果に手順・フロー・関係性などが含まれ図解が理解を助ける場合は、Mermaid図（フローチャート・シーケンス図など）の追加を提案し、含めるかどうかを同じ番号リストでまとめて確認する。
保存先パスなど他に未確定の項目がある場合は、同じ番号リストでまとめて確認する。
WAIT_FOR: ユーザーの確認。修正・追加調査の指示があれば反映してから先に進む（最大3ラウンド。収束しない場合は番号リスト形式でユーザーに方向性を確認し、指示があれば再開、なければ中断する）。

#### Mermaid 図を含める場合

- **HTML**: `<head>` に `<script src="https://cdn.jsdelivr.net/npm/mermaid@10/dist/mermaid.min.js"></script>` を追加し、本文に `<div class="mermaid">...</div>` で図を記述する。出力用 `<script>` に `mermaid.initialize({startOnLoad:true, theme:'neutral'})` を含める。
- **Markdown**: ` ```mermaid ` フェンスドコードブロックで記述する（GitHub/GitLab 等のビューアでネイティブ表示される）。
- 図の種類（flowchart / sequenceDiagram / classDiagram 等）は内容に応じて選択する。
- **quadrantChart の文字エンコーディング**: title・axis・quadrant・ポイント名に日本語等のUnicode文字を使う場合はダブルクォートで囲む（`"テキスト"`）。囲まないと `Syntax error in text` になる（flowchart 等の他の図種では不要）。
- **HTML でのキャプション**: 図の読み方や補足を添える場合は `<div class="mermaid">` の直後に `<span class="mermaid-caption">補足テキスト</span>` を置く。CSS は全テーマに組み込み済み。

#### HTML の場合

レイアウト選択 → カラー選択の **2 段階** で行う。

**ステップ 2-1: レイアウト選択**

まず各レイアウトのプレビュー画像（`light` カラー）を `Read` ツールで読み込んで表示し、視覚的にイメージを伝える:
- `$SKILL_DIR/assets/previews/simple-list-light.png`
- `$SKILL_DIR/assets/previews/panel-light.png`
- `$SKILL_DIR/assets/previews/article-light.png`
- `$SKILL_DIR/assets/previews/table-light.png`
- `$SKILL_DIR/assets/previews/slide-light.png`
IF 画像が読み込めない場合は表示をスキップし、テキストでレイアウトの特徴を説明する。

次に番号リストでレイアウトを選んでもらう:
- `simple-list` — 一覧・縦。テキスト中心・情報量が多いコンテンツ向き
- `panel` — 一覧・カード。同程度の重要度の項目を視覚的に並べる向き
- `article` — 文書。見出し・本文・コード・引用など Markdown 的な構造向き
- `table` — 表。複数項目を列で比較するデータ表向き
- `slide` — プレゼン資料。1セクション1枚の枠付きカードを横スクロール（矢印キー可）で送るスライド形式。提案・報告・ピッチ向き

**ステップ 2-2: カラー選択**

レイアウトが決まったら、ステップ2-1で選択した layout 値を `<layout>` に代入して全 6 色プレビューを `Read` ツールで読み込んで表示する:
- `$SKILL_DIR/assets/previews/<layout>-warm.png`
- `$SKILL_DIR/assets/previews/<layout>-cool.png`
- `$SKILL_DIR/assets/previews/<layout>-blue.png`
- `$SKILL_DIR/assets/previews/<layout>-green.png`
- `$SKILL_DIR/assets/previews/<layout>-dark.png`
- `$SKILL_DIR/assets/previews/<layout>-light.png`
IF 画像が読み込めない場合は表示をスキップし、テキストでカラーテーマの特徴を説明する。

次に番号リストで6色すべてを一度に提示してカラーテーマを選んでもらう:
```
[1]: warm — オレンジ・ベージュ。温かみがある
[2]: cool — パープル・ラベンダー。落ち着いた印象
[3]: blue — 青系。清潔感・信頼感
[4]: green — 緑系。自然・安心感
[5]: dark — 暗い背景。モダン・集中感
[6]: light — 白ベース。シンプル・クリーン
```

全 30 種のサンプルは `$SKILL_DIR/assets/samples/<layout>-<color>.html` に保存されている。
`open "$SKILL_DIR/assets/samples/<layout>-<color>.html"` でブラウザから詳細確認できることをユーザーに伝える。

**カスタマイズ範囲**

ステップ2-1で選択した layout 値とステップ2-2で選択した color 値を代入した
`$SKILL_DIR/assets/samples/<layout>-<color>.html` を Read してサンプルの構造を参照し、以下の範囲でカスタマイズしてよい:
- フォントファミリー・サイズ・行間
- 余白（padding / margin）
- アイコン・装飾（リストマーカー・ボーダー・シャドウ）
- セクション構成（追加・削除・並び替え）

**PROHIBITED: レイアウト骨格（グリッド・サイドバー有無・ヘッダー構造）を変更すること。**
**PROHIBITED: カラーテーマのパレット（色相・彩度）を大幅に変えること（アクセントカラーの微調整のみ許容）。**

#### Markdown の場合

番号リストで以下を確認する:

- **見出し構造**: フラット（H2 のみ）/ 階層あり（H2 + H3）
- **テーブル**: あり / なし
- **コードブロック**: あり / なし

#### CSV の場合

番号リストで以下を確認する:

- **区切り文字**: カンマ / タブ
- **ヘッダ行**: あり / なし
- **文字コード**: UTF-8 / Shift-JIS

#### JSON の場合

番号リストで以下を確認する:

- **構造**: 配列 / オブジェクト（キーあり）
- **インデント**: あり（2 スペース）/ なし（minified）

#### その他のフォーマット

形式・主要構成（セクション・項目）を列挙して確認する。不明点は 1 回でまとめて質問する。

### 3. 承認・修正

ユーザーの承認を求める前に、選択内容を以下の形式で要約して提示する:

```
- 形式: <HTML / Markdown / CSV / ...>
- レイアウト: <layout>（HTML の場合）
- カラー: <color>（HTML の場合）
- 図解: Mermaid図 <あり/なし>（含める場合）
- 保存先: <ファイルパス / チャット表示>
```

WAIT_FOR: ユーザーの承認（「OK」「はい」「進めて」等）を受けてからステップ4へ進む。
**PROHIBITED: 承認なしにステップ4（出力・保存）を実行すること。**
修正を求められた場合は、変更したい項目（レイアウト / カラー / 形式 / 保存先）を確認してから該当するステップに戻って再提示する（最大3ラウンド。収束しない場合は番号リスト形式でユーザーに方向性を確認し、指示があれば再開、なければ中断する）。

### 4. 出力

ステップ2で確認した形式・レイアウト・カラー等の選択を参照してアウトプットを生成する。

- ファイル保存の場合:
  1. `Bash`（`test -e <path>` 等）で既存ファイルの有無を確認し、存在する場合は上書きするか別名にするかをユーザーに確認する
     WAIT_FOR: ユーザーの選択
     **PROHIBITED: 確認なしに既存ファイルを上書きすること**
     別名を選んだ場合: 新しい保存先パスをユーザーに確認してから保存を続行する
  2. Write ツールで保存し、ファイルの存在を確認してからパスを報告する
  3. HTML の場合は「ブラウザで開きますか？」と確認する
     WAIT_FOR: ユーザーの承認
     承認されたら `Bash` で OS に応じたコマンドを実行する
     （macOS: `open <path>` / Linux: `xdg-open <path>` / Windows: `start <path>`）
- チャット表示の場合: コードブロックで出力する
- 保存に失敗した場合: エラーを報告し、別のパスをユーザーに確認する
- `RESEARCH_FILE` が存在する場合: `Bash` で `rm -f "$RESEARCH_FILE"` を実行して一時ファイルを削除する
