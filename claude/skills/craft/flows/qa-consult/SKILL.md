# qa-consult（QA相談フロー）

テストコードの実装を伴わない、QAに関する相談に対応する。
会話の途中で実装（テストコード追加・フレームワーク導入・CI組み込み等）が必要だと判明した場合は、
`{SKILL_DIR}/flows/qa/SKILL.md` に切り替える。

```
SKILL_DIR = このSKILL.mdが存在するディレクトリの2階層上の絶対パス
  # このファイルを /Users/alice/.claude/skills/craft/flows/qa-consult/SKILL.md で読んだ場合
  # → SKILL_DIR = /Users/alice/.claude/skills/craft
```

---

## 手順

### ステップ1: 相談種別の判定

```
IF consult から SUMMARY（ここまでの相談内容の要約）が渡されている:
  USE: SUMMARY を種別推定の材料として使う

キーワード例:
  1（テスト計画・戦略立案） → テスト計画・テストケース・テスト観点・カバレッジ・リスクベース
  2（QA体制・プロセス構築） → 体制・組織・標準化・テンプレート・属人化・アジャイルQA
  3（品質指標・バグ管理）   → バグ密度・欠陥・トリアージ・重大度・優先度・品質指標
  4（横断的な相談）        → 上記に当てはまらない、または複数にまたがるQA相談

IF 種別が明確に推定できる:
  CONFIRM: 「〇〇（種別名）の相談として進めますか？」と一言確認する
  WAIT_FOR: ユーザーの回答
  IF ユーザーが否定した:
    ASK USER: どの種別か（上記1〜4から選択）
    WAIT_FOR: ユーザーの回答
ELSE:
  ASK USER: どの種別か（上記1〜4から選択）
  WAIT_FOR: ユーザーの回答
```

### ステップ2: 種別ごとの観点整理

ステップ1で判定した種別番号に対応するセクションのみを適用する（4は1〜3を横断する）。

#### 1. テスト計画・戦略立案

- テストレベルを整理する（コンポーネント/単体・結合・システム・受入）
- テスト観点の網羅性をチェックする（機能・非機能・セキュリティ・性能・データ）
- リスクベースで優先順位をつける（影響度 × 発生確率が高い領域を重点化する）
- シフトレフト（開発初期からの早期検出）・シフトライト（本番環境でのモニタリング）の適用余地を検討する

#### 2. QA体制・プロセス構築

- 開発ライフサイクル全体（計画・設計・実装・テスト・リリース）のどの段階にQAが関与するかを設計する
- 属人化を避けるため、テスト観点・手順・ドキュメントを標準化・テンプレート化する
- チーム体制（専任QA・開発兼任・外部委託など）と役割分担を整理する

#### 3. 品質指標・バグ管理レビュー

- バグ密度（バグ数 / 規模）などの指標を算出し、プロダクト品質とプロセス品質を分けて評価する
- Severity（重大度）と Priority（優先度）のマトリクスで対応順序を可視化する
- バグトリアージのプロセス（受付 → 再現確認 → 優先度付け → アサイン →検証）を設計する

#### 4. 横断的な相談

上記1〜3の観点を組み合わせ、状況に応じて相談に応じる。

### ステップ3: まとめ・次のアクション

```
REPORT: 整理した観点・推奨事項をユーザーに提示する

GATE: この内容で十分か、追加の深掘りが必要かをユーザーに確認する
  → 追加の深掘りが必要な場合はステップ2の該当観点に戻る

IF 相談の結果、テストコードの実装・フレームワーク導入・CI組み込みが必要と判明:
  PROPOSE: 「実装フェーズに進む場合は qa フローに切り替えます」と提案する
  WAIT_FOR: ユーザーの回答
  IF ユーザーが承認:
    SUMMARY = この相談で整理した内容の要約（種別・観点・推奨事項）
    READ {SKILL_DIR}/flows/qa/SKILL.md
    FOLLOW: そこに記述されたすべての手順を実行する。
            ステップ1（ブリーフィング）には SUMMARY を渡し、不足している点のみ質問させる。
ELSE:
  STOP: 相談・提案の提示をもって完了とする
```
