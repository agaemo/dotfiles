# レガシーコードへの安全なテスト追加

目的: 構造を変える前に「今の動作」を固定するセーフティネットを作ってから着手する。
原則: **Characterize → Seam** の順を守る（テストなしに構造を変えない）。例外は判定Cのみ。

```
STEP A: テスト可否を判定する
  判定A: そのままテストが書ける → STEP B へ
  判定B: 結合度が高いがモジュールはインポートできる → 統合テスト・E2Eで代替（STEP B へ）
  判定C: モジュールのインポート自体が失敗する（フレームワーク初期化が実行環境を要求する等）
    → STEP C（Seam抽出）を先に行い、完了後に STEP A の判定から再実行する

STEP B: Characterization Test を書く（既存動作の記録。「正しさ」は問わない）
  - 代表的な入力・エッジケースでコードを読み、実際の出力・副作用を書き出す
  - 「今こう動いている」をそのまま期待値にする（予想で書くと Red になる）
  - 実行して Green になることを確認する（最初から Green になるはず）
    Red になった場合: 実際の挙動を読み直して期待値を修正する。3回以上続く場合はユーザーに報告する
  - Green確認後、git commit する（この commit が STEP C のリファクタ中の安全網になる）

STEP C: Seam（縫い目）を切り出す（判定B/Cで構造変更が必要な場合のみ。ロジックは変えない）
  以下のパターンがないか確認する:
    1. フレームワーク依存の中にビジネスロジックが埋まっている
    2. 外部依存（DB・API・ファイル・時刻・乱数）を直接呼んでいる
    3. グローバル状態・シングルトンへの直接アクセスがある
    4. モジュール初期化時の副作用でインポートが失敗する（判定Cの原因）

  FOR EACH 発見したパターン:
    PROPOSE: 切り出し方法（ビジネスロジックを純粋関数として切り出す／外部依存を引数として注入する）を提示する
    WAIT_FOR: ユーザーの承認
    REFACTOR: 元のコードはその純粋関数を呼ぶ薄いグルーコードにする
    VERIFY: ビルドが通り、STEP B の Characterization Test が全件 Green のままであること
    IF Green が崩れて修正できない場合: git reset で STEP B の commit まで戻し、ユーザーに報告する

  ADD TESTS: 切り出した純粋関数に tester エージェント（補完モード）でユニットテストを追加する
```
