---

# migrate（移行フロー）

既存プロジェクトの構造的な変更を、ブリーフィング → 現状調査 → 計画 → 段階的実行の流れで安全に進める。

---

## 手順

### ステップ 1: ブリーフィング

```
ASK USER:
  どのような移行を検討していますか？
  現状と目指す姿、背景を教えてください。

WAIT_FOR: ユーザーの説明

EXTRACT:
  FROM        = 現在の技術・構成（例: SQLite、自前 JWT 認証）
  TO          = 目標の技術・構成（例: MySQL、Auth.js）
  REASON      = 理由・背景（例: ユーザー増加、セキュリティ要件）
  CONSTRAINTS = 制約（ダウンタイム許容可否・本番稼働中か・期限等）

IF 情報が不足している場合:
  ASK: 不明点のみを1件ずつ質問する（一問一答・まとめて聞かない）
  WAIT_FOR: 回答
  REPEAT: すべての必須情報が揃うまで繰り返す
```

### ステップ 2: 現状調査

```
READ: コードベースを調査し以下を把握する
  - FROM に関連するファイル・設定・依存パッケージ
  - 影響を受けるモジュール・API・テストの一覧
  - テストカバレッジの有無
  - 環境変数・設定ファイルの依存箇所

REPORT: 調査結果をユーザーに提示
  - 影響ファイル数と主要な変更箇所
  - 懸念点（テスト不足・複雑な依存・ダウンタイムリスク等）
```

### ステップ 3: 移行計画の策定・承認

```
PLAN:
  移行をフェーズに分割する。各フェーズは以下の条件を満たすこと：
    - 独立してテスト・検証できる単位
    - ロールバック手順が明確
    - 前のフェーズが成功しないと次に進まない

  計画フォーマット：
    ## 移行計画: FROM → TO

    ### フェーズ 1: [名称]
    - 変更内容: ...
    - 変更ファイル: ...
    - 検証方法: ...
    - ロールバック: git revert / 手順

    ### フェーズ 2: ...

    ### 残作業（手動対応が必要なもの）
    - ...

PRESENT: 計画をユーザーに提示する
GATE 1/N: 計画の承認を得る
  → 承認されるまで実行しないこと
  → 修正要望があれば計画を更新して再提示する
```

### ステップ 4: 段階的実行

```
BRANCH = migration/<from>-to-<to>
  例: migration/sqlite-to-mysql、migration/jwt-to-authjs

git checkout -b BRANCH

FOREACH phase IN 計画:
  REPORT: 「フェーズ N を開始します」と通知

  EXECUTE: そのフェーズの変更を適用する

  VERIFY:
    IF validateCommand が設定されている（.forager/config.json 等）:
      validateCommand を実行する
    ELSE:
      ビルド・lint・型チェックのうち利用可能なものを実行する

  IF 検証失敗:
    ROLLBACK: そのフェーズの変更を元に戻す（git checkout -- .）
    REPORT: 失敗内容と原因をユーザーに報告する
    STOP: ユーザーの判断を仰ぐ

  GATE: 次フェーズへの進行承認を得る
    → フェーズ完了を報告し、続行してよいか確認する
```

### ステップ 5: 完了・PR 作成

```
SUMMARY:
  - 変更したファイル一覧
  - 削除した旧コード・依存パッケージ
  - 追加した新しい依存パッケージ
  - 残作業（設定値の本番反映・環境変数の更新等、手動対応が必要なもの）

CREATE PR:
  タイトル: migrate: FROM → TO（例: migrate: SQLite → MySQL）
  本文:
    ## 移行概要
    [背景と目的を1〜2文で]

    ## 変更内容
    [フェーズごとの変更概要]

    ## 残作業
    [手動で対応が必要なことのチェックリスト]

    ## テスト
    [検証コマンドの実行結果]
```
