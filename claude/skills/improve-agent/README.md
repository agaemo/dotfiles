# improve-agent

エージェントファイル（`.claude/agents/*.md`）を静的解析・実行シミュレーションで改善するスキル。
実プロジェクトを動かさずに低トークンで反復改善できる。

## 使い方

```
/improve-agent <ファイルパス>        # 単体
/improve-agent path/a.md path/b.md  # 複数
/improve-agent all                  # .claude/agents/ 全件
```

## 6次元の静的評価

各次元を 1〜5 で採点し、改善提案を生成する。

| 次元 | 観点 | 何をチェックするか |
|------|------|--------------------|
| **[A] トークン効率** | コスト | ロール・プロセス・ルール間での同内容の重複。削除しても意味が変わらない前置き・まとめ。ルールがプロセスの言い換えになっていないか。 |
| **[B] 役割の純粋性** | スコープ | 「何をしないか」が明記されているか。他エージェントの担当領域（例: planner が担うべき設計）を侵食していないか。完了報告で次エージェントへ適切に委任しているか。 |
| **[C] 命令の明瞭性** | 解釈ブレ | 各ステップが1通りにしか解釈できないか。「適切に」「必要に応じて」等の曖昧な副詞がないか。ステップ間のデータ受け渡しが暗黙になっていないか。ループの粒度（ファイルごと vs 全体で1回）が明示されているか。 |
| **[D] tools/model 適切性** | 設定 | 使っていない可能性が高いツールが列挙されていないか。model の選択（haiku / sonnet / opus）が役割の複雑さに見合っているか。目的を達成するために必要なツールが不足していないか。 |
| **[E] 出力設計** | 出力品質 | 出力フォーマットが定義されているか（フリーテキストになっていないか）。成功・失敗・スキップ時の出力が区別されているか。下流エージェントが読むファイルのパスが明記されているか。 |
| **[F] 検証可能性** | 自己確認 | エージェントが作業完了を自己判断できる基準があるか（テスト実行・ファイル確認等）。重要なルールに `IMPORTANT` / `YOU MUST` 等の強調が付いているか。不可逆・高コストな操作の前に確認ステップがあるか。 |

## Adversarial Trace

「コールドスタートの Claude がこのエージェントを初めて実行する」ペルソナで、
意図的に誤解しやすい解釈でステップを読み、スコープ逸脱・手抜き・誤動作が起きる箇所を探す。

## 評価後のフロー

1. スコアと改善提案を表示
2. 全書き直し判定（2/5 以下が 3 次元以上ならユーザーに確認）
3. ユーザーが適用する提案番号を選択（`all` / `1 3 5` / `skip`）
4. 適用 → READ で確認 → 報告
