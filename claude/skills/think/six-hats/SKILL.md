# Six Thinking Hats — 6視点並列分析

Edward de Bono (1985) の Six Thinking Hats を6エージェントで実装する。
トピック・提案書・意思決定を渡すと、5エージェントが並列分析し、青帽がプロセスを整理する。

```
SKILL_DIR = スキル起動時に提供される "Base directory for this skill:" のパス
```

## このスキルでできること・できないこと

**向いている入力:**
- 具体的な提案・計画・アイデアの多角的検証（例：「新製品Xを日本市場に投入する計画」）
- 2択以上の選択肢の比較検討（例：「AかBか、どちらのアーキテクチャを採用するか」）
- 意思決定前の論点整理（例：「スタートアップに参画するかどうか」）

**向いていない入力（受け付けない）:**
- 前提条件が未定義の広い問い（例：「何かいいビジネスはないか」「次に何をすべきか」）
- 「答えを1つ出してほしい」という要求 — Six Hatsは答えを導くツールではなく、人間の判断を支援するツール（de Bono, 1985）
- 事実調査・検索が必要な問い

---

## 引数の処理

- **引数あり**: まず入力の適切性を判定する（下記参照）
- **引数なし**: 「何を分析するか?」と一言聞く。ユーザーの返答を分析対象とする
- **`{verbosity}`**: think から渡される。渡されていない場合は 標準 とする

### 入力の適切性判定

入力を受け取ったら分析を始める前に以下を判定する。

**NG判定の条件（1つでも該当したら断る）:**
1. 「何かいい〇〇はないか」「〇〇を探している」のように、答え自体の生成を求めている
2. ターゲット・制約・前提条件が何も含まれておらず、問いが広すぎる
3. 「〇〇を教えて」のように情報検索が目的

NGの場合は分析を開始せず、以下を返す：
```
このスキルは「具体的な提案・計画・選択肢」の多角的検証に向いています。
今の入力「{input}」は[NG理由]のため、Six Hatsでの分析には向いていません。

分析するには入力を具体化してください。例：
- 対象を絞る（誰向け？どの市場？）
- 選択肢を持ち込む（A案 vs B案）
- 既存の計画・提案を渡す
```

OK判定の場合のみ、以降のステップに進む。

以降、分析対象を `{input}` と呼ぶ。

---

## ブリーフィング（補足情報収集）

分析精度を高めるために補足情報を収集する。以下の質問を出力してから WAIT_FOR: ユーザーの回答。
「スキップ」「省略」と返答された場合は `{context}` = "" としてステップ1に進む。

1. **提案・計画の詳細**: 背景・目的・規模感を補足してください（ステークホルダー・予算・期間など）
2. **特に気になっている観点**: すでにリスクや懸念として頭にあるものがあれば
3. **意思決定の制約**: 期限・意思決定者・覆せない前提条件などがあれば

ユーザーの回答を `{context}` として保持し、変数 `{enriched_input}` を定義する:
- `{context}` が空でない場合: `{enriched_input}` = `{input}` の末尾に `\n\n## 補足情報\n{context}` を追加したもの
- `{context}` が空の場合: `{enriched_input}` = `{input}`

IMPORTANT: 以降のすべてのエージェント起動では `{input}` の代わりに `{enriched_input}` を使うこと。対話フェーズのagent再起動でも同様。

---

## ステップ1: 5エージェントを並列起動

以下のファイルを READ し、ファイル内の `{input}` を分析対象テキストに置換したプロンプトとして `run_in_background: true` で**同時に** Agent 起動する。全エージェントの完了通知を受け取ってからステップ2に進む。

- READ {SKILL_DIR}/agents/white.md → {enriched_input} を置換して Agent 起動（白帽: 事実）
- READ {SKILL_DIR}/agents/black.md → {enriched_input} を置換して Agent 起動（黒帽: リスク）
- READ {SKILL_DIR}/agents/yellow.md → {enriched_input} を置換して Agent 起動（黄帽: 価値）
- READ {SKILL_DIR}/agents/red.md → {enriched_input} を置換して Agent 起動（赤帽: 直感）
- READ {SKILL_DIR}/agents/green.md → {enriched_input} を置換して Agent 起動（緑帽: 代替案）

エージェントが失敗した場合は「報告なし」として扱い、青帽の統合で欠損を明記する。

---

## ステップ2: 青帽による統合・出力

5エージェントの結果を受け取り、以下の内容を組み立てる。

### verbosity に応じた出力

| verbosity | 会話上の出力 | ファイル保存 |
|---|---|---|
| 簡潔 | サマリー表のみ | なし |
| 標準 | サマリー表 + 青帽の整理 | 保存先を聞く |
| 詳細 | サマリー表 + 青帽の整理 + 各帽子の詳細 | 保存先を聞く |

PROHIBITED: ユーザーの保存先確認なしにファイルを書くこと

verbosity = 簡潔 の場合: ファイル保存は行わない。下記テンプレートの「## サマリー」形式で会話上に出力し、ステップ3（対話）に進む。
verbosity = 標準 / 詳細 の場合: 「保存先のパスを教えてください（例: ~/Desktop）」と一言聞く。ユーザーが答えたパスに対し:

1. Bash で `date +%Y%m%d-%H%M` を実行してファイル名用タイムスタンプを、`date +%Y-%m-%d` を実行してレポートタイトル用日付をそれぞれ取得する。ファイル名は `six-hats-<YYYYMMdd-HHmm>.md`（例: six-hats-20260530-1430.md）とする
2. 指定パスが存在しない場合は Bash で `mkdir -p <path>` を実行してから保存する
3. Write ツールで保存する

パスが省略されたり「ここ」「カレント」と言われた場合は `{SKILL_DIR}/../`（= think/ 直下）に保存する。

---

Write 直前に `{SKILL_DIR}/templates/report.md` を READ し、`{enriched_input}` の冒頭30字・`{YYYY-MM-DD}` および各帽子エージェントの出力を対応する変数に置換したものを保存内容とする。

---

ファイル保存後、会話上には「`{実際の保存フルパス}` に保存しました」と一言だけ伝える。
verbosity = 詳細 の場合は詳細セクションも会話上に出力する。それ以外は詳細セクションを会話上には出力しない。

---

## ステップ3: 対話

レポート出力後、以下を受け付ける。

- 「黒帽をもっと詳しく」など特定帽子への深掘り依頼 → {enriched_input}（補足情報を含む分析対象）を置換して該当エージェントを再起動し、補足を生成。結果は会話上のみに出力し、ファイルは更新しない
- 「別の案を出して」「緑帽に追加を」→ {enriched_input}（補足情報を含む分析対象）を置換して緑帽エージェントを再起動。結果は会話上のみに出力し、ファイルは更新しない
- 「〇〇の方針で進める場合のリスクは」→ {enriched_input}（補足情報を含む分析対象）を置換して黒帽エージェントを再起動し、特定シナリオのリスクを生成。結果は会話上のみに出力し、ファイルは更新しない
- 「終わり」→ 簡潔に締める
- その他の質問・発話 → 青帽として統合的な視点で答える
