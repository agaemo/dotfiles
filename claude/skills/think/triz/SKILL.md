# TRIZ — 発明原理による矛盾解消

Genrich Altshuller が40万件のソ連特許を分析して開発した体系的発明方法論（1956〜）。
「AとBを同時に満たせない」という矛盾に対して、40の発明原理から解決の方向性を引き出す。

```
SKILL_DIR = スキル起動時に提供される "Base directory for this skill:" のパス
```

## このスキルでできること・できないこと

**向いている入力:**
- 「速くするとコストが上がる」「軽くすると強度が落ちる」のようなトレードオフ
- 「〇〇でありながら〇〇でもある」ことを同時に求められている矛盾
- 複数の要件が構造的に相反していて、妥協以外の解法を探している

**向いていない入力（受け付けない）:**
- まだ矛盾が発見されていない漠然とした問い（まず問題を具体化してほしい）
- 既存アイデアの変形・改良（→ scamper が向いている）
- 前提を疑うゼロベース発想（→ first-principles が向いている）
- 多角的検証・比較（→ six-hats が向いている）

---

## 引数の処理

- **引数あり**: 入力の適切性を判定する（下記参照）
- **引数なし**: 「どのような矛盾・トレードオフに直面していますか? 例：『速くしたいが、コストは上げられない』」と一言聞く
- **`{verbosity}`**: think から渡される。渡されていない場合は 標準 とする

### 入力の適切性判定

**NG判定の条件（1つでも該当したら断る）:**
1. 矛盾・トレードオフの対象が何も含まれていない
2. 既存の改良や比較検討が目的（→ scamper / six-hats）
3. 「情報を教えてほしい」という情報収集が目的

NGの場合:
```
TRIZ は「相反する要件を同時に満たす」解決策を探す手法です。
今の入力「{input}」には解消すべき矛盾・トレードオフが明確でないため、TRIZ には向いていません。

「〇〇すると△△が悪化する」「〇〇でありながら〇〇でもある必要がある」のように、
矛盾を含む問題を教えてください。
```

OK判定の場合のみ以降に進む。以降、分析対象を `{input}` と呼ぶ。

---

## ブリーフィング（補足情報収集）

分析精度を高めるために補足情報を収集する。以下の質問を出力してから WAIT_FOR: ユーザーの回答。
「スキップ」「省略」と返答された場合は `{context}` = "" としてステップ1に進む。

1. **矛盾の具体的な状況**: 数値・具体例で教えてください（例: 処理速度を上げると消費電力が3倍になる）
2. **試みた解決策とその限界**: これまで検討・実施したアプローチと、なぜうまくいかなかったか
3. **要件の優先度**: どちらの要件をより優先しなければならないか（あれば）

ユーザーの回答を `{context}` として保持し、変数 `{enriched_input}` を定義する:
- `{context}` が空でない場合: `{enriched_input}` = `{input}` の末尾に `\n\n## 補足情報\n{context}` を追加したもの
- `{context}` が空の場合: `{enriched_input}` = `{input}`

IMPORTANT: 以降のすべてのエージェント起動では `{input}` の代わりに `{enriched_input}` を使うこと。対話フェーズのagent再起動でも同様。

---

## ステップ1: 矛盾の特定（フェーズ1）

`{SKILL_DIR}/agents/contradiction-finder.md` を READ し、`{enriched_input}` を置換して Agent 起動する。
完了を待ってから、contradiction-finder の出力末尾「最も重要な矛盾（次フェーズの入力）:」セクション以降のテキストを `{contradiction}` として保持し、ステップ2に進む。
エージェントが失敗した場合は「矛盾の分析を完了できませんでした。再度お試しください。」と伝えてSTOP。

---

## ステップ2: 発明原理の適用（フェーズ2）

`{SKILL_DIR}/agents/principle-matcher.md` を READ し、`{enriched_input}` と `{contradiction}` を置換して Agent 起動する。
完了を待ってからステップ3に進む。
エージェントが失敗した場合は「発明原理の適用を完了できませんでした。再度お試しください。」と伝えてSTOP。

---

## ステップ3: 統合・出力

2フェーズの結果を受け取り、以下の内容を組み立てる。

### verbosity に応じた出力

| verbosity | 会話上の出力 | ファイル保存 |
|---|---|---|
| 簡潔 | 最も有望な解決案のみ | なし |
| 標準 | サマリー全体（矛盾 + 原理一覧 + 有望案） | 保存先を聞く |
| 詳細 | サマリー + 全フェーズ詳細 | 保存先を聞く |

PROHIBITED: ユーザーの保存先確認なしにファイルを書くこと

verbosity = 簡潔 の場合: ファイル保存は行わない。下記テンプレートの「## サマリー」セクションのみ会話上に出力し、ステップ4（対話）に進む。
verbosity = 標準 / 詳細 の場合: 「保存先のパスを教えてください（例: ~/Desktop）」と一言聞く。ユーザーが答えたパスに対し:

1. Bash で `date +%Y%m%d-%H%M` を実行してタイムスタンプを、`date +%Y-%m-%d` を実行して日付をそれぞれ取得する。ファイル名は `triz-<YYYYMMdd-HHmm>.md` とする
2. 指定パスが存在しない場合は Bash で `mkdir -p <path>` を実行してから保存する
3. Write ツールで保存する
4. Bash で `test -f <保存フルパス> && echo OK` を実行しファイルの存在を確認する。失敗した場合は Write を再試行する（最大1回）。

パスが省略されたり「ここ」「カレント」と言われた場合は `{SKILL_DIR}/../`（= think/ 直下）に保存する。

---

Write 直前に `{SKILL_DIR}/templates/report.md` を READ し、`{input}` `{YYYY-MM-DD}` `{contradiction}` および各フェーズエージェントの出力を対応する変数に置換したものを保存内容とする。

---

ファイル保存後、会話上には「`{実際の保存フルパス}` に保存しました」と一言伝える。
verbosity = 詳細 の場合は詳細フェーズも会話上に出力する。それ以外は詳細フェーズを会話上には出力しない。

---

## ステップ4: 対話

- 「別の原理を試してほしい」→ principle-matcher を `{contradiction}` で再起動して別の原理を適用。結果は会話上のみ出力
- 「この解決案のリスクは?」→ six-hats での検証を提案
- 「終わり」→ 簡潔に締める
- その他 → 統合した視点で答える
