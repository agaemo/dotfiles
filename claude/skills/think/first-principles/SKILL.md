# First Principles — 第一原理思考

アリストテレスが定式化し、Elon Musk が現代ビジネスに持ち込んだ推論法。
「当たり前」を全て疑い、変えられない事実だけから出発してゼロから再構築する。

```
SKILL_DIR = スキル起動時に提供される "Base directory for this skill:" のパス
```

## このスキルでできること・できないこと

**向いている入力:**
- 「なぜこんなにコストがかかるのか」「なぜこの順序でなければならないのか」という根本的疑問
- 業界の常識・慣習を疑い、全く新しいアプローチを探したい
- 既存のやり方が前提となっていて行き詰まっている問題

**向いていない入力（受け付けない）:**
- 「既存のXをもっと良くしたい」（変形・改良 → scamper が向いている）
- 「AとBのどちらが良いか」（比較検証 → six-hats が向いている）
- 「相反する要件を両立したい」（矛盾解消 → triz が向いている）

---

## 引数の処理

- **引数あり**: 入力の適切性を判定する（下記参照）
- **引数なし**: 「どんな問題・状況について前提を疑いたいですか?」と一言聞く
- **`{verbosity}`**: think から渡される。渡されていない場合は 標準 とする

### 入力の適切性判定

**NG判定の条件（1つでも該当したら断る）:**
1. 前提を疑うべき対象・問題が何も含まれていない
2. すでに解決策が決まっており、比較・評価を求めている
3. 「改良・改善」のみが目的（既存を活かしたい → scamper）

NGの場合:
```
第一原理思考は「なぜそうしているのか」を根本から問い直す手法です。
今の入力「{input}」には疑うべき前提・問題が明確でないため、このスキルには向いていません。

「なぜ〇〇はこういう仕組みなのか?」「〇〇のコストがなぜ高いのか?」のように、
疑問の対象を具体的に教えてください。
```

OK判定の場合のみ以降に進む。以降、分析対象を `{input}` と呼ぶ。

---

## ブリーフィング（補足情報収集）

分析精度を高めるために補足情報を収集する。以下の質問を出力してから WAIT_FOR: ユーザーの回答。
「スキップ」「省略」と返答された場合は `{context}` = "" としてステップ1に進む。

1. **「当たり前」の由来**: 疑いたい常識・慣習はいつ頃・なぜ定着したか（知っている範囲で）
2. **変えられない制約**: 物理的限界・法規制・安全要件など、どう考えても変えられないものは
3. **再構築のタイムライン**: 理想の実現はどのくらいのスパンを想定しているか（例: 1年以内・5年以上）

ユーザーの回答を `{context}` として保持し、変数 `{enriched_input}` を定義する:
- `{context}` が空でない場合: `{enriched_input}` = `{input}` の末尾に `\n\n## 補足情報\n{context}` を追加したもの
- `{context}` が空の場合: `{enriched_input}` = `{input}`

IMPORTANT: 以降のすべてのエージェント起動では `{input}` の代わりに `{enriched_input}` を使うこと。対話フェーズのagent再起動でも同様。

---

## ステップ1: 前提ハンター（フェーズ1）

`{SKILL_DIR}/agents/assumption-hunter.md` を READ し、`{enriched_input}` を置換して Agent 起動する。
完了を待ってから、エージェントの出力全文を `{assumptions}` として保持し、ステップ2に進む。
失敗した場合は「前提の分析を完了できませんでした。再度お試しください。」と伝えてSTOP。

---

## ステップ2: 前提の検証（フェーズ2）

`{SKILL_DIR}/agents/challenger.md` を READ し、`{enriched_input}` と `{assumptions}` を置換して Agent 起動する。
完了を待ってから、challenger の出力中以下のセクションをそれぞれ抽出してステップ3に進む:
- `{true_constraints}`: 「絶対に変えられない制約（真）:」セクション以降の箇条書き全文
- `{challengeable_assumptions}`: 「覆せる可能性が高い前提（慣習・要検証）:」セクション以降の箇条書き全文
失敗した場合は「前提の検証を完了できませんでした。再度お試しください。」と伝えてSTOP。

---

## ステップ3: ゼロから再構築（フェーズ3）

`{SKILL_DIR}/agents/rebuilder.md` を READ し、`{enriched_input}`・`{true_constraints}`・`{challengeable_assumptions}` をそれぞれ対応するテキストに置換して Agent を起動する。
完了を待ってからステップ4に進む。
失敗した場合は「ゼロからの再構築を完了できませんでした。再度お試しください。」と伝えてSTOP。

---

## ステップ4: 統合・出力

3フェーズの結果を受け取り、以下の内容を組み立てる。

### verbosity に応じた出力

| verbosity | 会話上の出力 | ファイル保存 |
|---|---|---|
| 簡潔 | サマリー（制約・覆せた前提・理想形）のみ | なし |
| 標準 | サマリー全体 | 保存先を聞く |
| 詳細 | サマリー + 全フェーズ詳細 | 保存先を聞く |

PROHIBITED: ユーザーの保存先確認なしにファイルを書くこと

verbosity = 簡潔 の場合: ファイル保存は行わない。下記テンプレートの「## サマリー」セクションのみ会話上に出力し、ステップ5（対話）に進む。
verbosity = 標準 / 詳細 の場合: 「保存先のパスを教えてください（例: ~/Desktop）」と一言聞く。ユーザーが答えたパスに対し:

1. Bash で `date +%Y%m%d-%H%M` を実行してタイムスタンプを、`date +%Y-%m-%d` を実行して日付をそれぞれ取得する。ファイル名は `first-principles-<YYYYMMdd-HHmm>.md` とする
2. 指定パスが存在しない場合は Bash で `mkdir -p <path>` を実行してから保存する
3. Write ツールで保存する
4. Bash で `test -f <保存フルパス> && echo OK` を実行しファイルの存在を確認する。失敗した場合は Write を再試行する（最大1回）。

パスが省略されたり「ここ」「カレント」と言われた場合は `{SKILL_DIR}/../`（= think/ 直下）に保存する。

---

Write 直前に `{SKILL_DIR}/templates/report.md` を READ し、`{input}` `{YYYY-MM-DD}` `{true_constraints}` `{challengeable_assumptions}` および各フェーズエージェントの出力を対応する変数に置換したものを保存内容とする。

---

ファイル保存後、会話上には「`{実際の保存フルパス}` に保存しました」と一言伝える。
verbosity = 詳細 の場合は詳細フェーズも会話上に出力する。それ以外は詳細フェーズを会話上には出力しない。

---

## ステップ5: 対話

- 「フェーズ1をもっと詳しく」→ assumption-hunter.md を `{enriched_input}` 置換して再起動し、補足を会話上のみ出力
- 「フェーズ2をもっと詳しく」→ `{enriched_input}` と `{assumptions}` を渡して challenger.md を再起動し、補足を会話上のみ出力
- 「フェーズ3をもっと詳しく」→ `{enriched_input}`・`{true_constraints}`・`{challengeable_assumptions}` を渡して rebuilder.md を再起動し、補足を会話上のみ出力
- 「このアプローチで進めた場合のリスクは?」→ six-hats での検証を提案
- 「終わり」→ 簡潔に締める
- その他 → 統合した視点で答える
